# BBF Captcha – Ist-Stand & Befunde

Stand: 2026-06-09 · Version 1.0.0 · MinShopVersion 5.2.0

Grundlage für den führenden [Masterplan](masterplan.md). Befunde sind am Code
verifiziert (Datei:Zeile). Bewertung folgt der Entwicklungssteuerung
(`docs/claude-development-control.md`): „Kunden nie aussperren", Fail-open,
Sekret-Dichtheit, DSGVO, Template-/Versionsrobustheit, Standalone-Vorrang.

## 1. Architektur in Kürze

- **Hooks** (`Bootstrap.php`): `HOOK_SMARTY_OUTPUTFILTER` (Honeypot/Timing in alle
  `<form>`), plus seitenspezifische Hooks für Kontakt, Registrierung, Newsletter,
  Bewertung, Checkout, Login, Wunschliste, Passwort-Reset; optionale
  Consent-Manager-Integration.
- **Injection**: `src/Hooks/IncludeAssets.php` (CSS/JS + Public-Sitekeys vor
  `</head>`), `src/Hooks/SmartyOutputFilter.php` (regexbasiert `<form\b…>`),
  `src/Hooks/FormProtection.php` (Smarty-Variable `$bbfCaptchaWidget`).
- **Provider/Services** (`src/Services/`): Honeypot, Timing, Altcha (lokal, PoW),
  Turnstile/reCAPTCHA/hCaptcha/Friendly (extern), RateLimit, IP, BotDetector,
  AISpam (lokal, Regeln), LLMSpam (extern, optional), Consent, SpamLog.
- **DB** (`Migrations/`): `bbf_captcha_settings`, `_form_config`, `_spam_log`,
  `_ip_entries`, `_rate_limits`, `_api_keys`, `_spam_words`, `_disposable_domains`.
- **Backend**: Dashboard, Einstellungen, Formularschutz, IP-Verwaltung, LLM,
  Log, API, Doku (`adminmenu/templates/*.tpl`).
- **Standalone**: keine Abhängigkeit zu anderen BBF-Plugins; Namespace
  `Plugin\bbfdesign_captcha\*`; Assets plugin-relativ. Gute Ausgangslage.

## 2. Verifizierte Befunde

### 2.1 Fail-open / „Kunden nie aussperren" (höchste Priorität)

- **Externe Provider ohne Fallback-Kaskade.** Bei Timeout/Netzfehler von
  Turnstile/reCAPTCHA/hCaptcha/Friendly wird ein Strafscore gesetzt, aber **nicht
  automatisch auf den lokalen Altcha-/Honeypot-Pfad ausgewichen**. Folge: echte
  Kunden können bei Provider-Ausfall an Login/Registrierung/Checkout hängen
  bleiben. → Fallback-Kaskade nötig (`src/Services/CaptchaService.php`,
  externe Provider-Services mit 10s/5s-Timeout).
- **LLM-Zweitprüfung im Absende-Hotpath.** `LLMSpamService` ruft synchron mit
  ~8s-Timeout; `failOpen()` existiert, aber der regelbasierte Restscore kann
  trotzdem blockieren. → LLM strikt non-blocking und rein additiv (nie allein
  blockierend), Ergebnis-Caching pro Text-Hash.
- **Template-Render-Fehler → leeres Widget.** Schlägt das Widget-Rendering fehl,
  wird im Zweifel nichts ausgeliefert statt eines sicheren Fallbacks
  (`FormProtection.php`, `IncludeAssets.php`).

### 2.2 Sicherheit / Sekret-Dichtheit

- **Custom-CSS roh in `<style>`.** `IncludeAssets.php:53–57` entfernt nur
  `</style`, injiziert den Admin-Wert sonst ungefiltert → CSS-Injection-Risiko
  (Exfiltration via `background:url(...)`, Layout-Sabotage). → Whitelist-Validator
  oder sichere Auslieferung.
- **Frontend secret-frei (gut).** Im Browser nur öffentliche Sitekeys; Secrets in
  `bbf_captcha_settings`. Der Secret-Scan des `--local`-Gates ist grün.
- **Altcha-HMAC-Key** wird nur bei Installation erzeugt (`Bootstrap.php`); keine
  Rotation per UI. → Rotations-Aktion mit Übergangsfenster.

### 2.3 DSGVO

- **Volle IP-Speicherung** in `bbf_captcha_spam_log` (`VARCHAR(45)`, 90 Tage),
  keine eingebaute Anonymisierung, Export potenziell ungefiltert. →
  Anonymisierungs-Option (IPv4 /24, IPv6 /64) + Export-Filter; Aufbewahrung
  konfigurierbar; Cleanup zuverlässig triggern (heute nur als Methode vorhanden).

### 2.4 Template-/Versionsrobustheit

- **Form-Erkennung ist regexbasiert und ohne harte Theme-Selektoren** (gut für
  NOVA/Fremdtemplate). Risiko: Formulare, die das Widget nicht über
  `{$bbfCaptchaWidget}` einbinden, bleiben ohne sichtbares Captcha — die
  automatische HTML-Marker-Ersetzung ist noch nicht aktiv (`Bootstrap.php`,
  Kommentar). → Zuverlässige, theme-unabhängige Auto-Platzierung.
- **Keine Versions-Guards**, genutzt werden nur stabile JTL-5-APIs
  (`Dispatcher`, `JTLSmarty`, `queryPrepared`). Betrieb 5.2–5.7 plausibel, aber
  **nicht über eine Matrix getestet**. → Test-Matrix, primär gegen 5.7.x.

### 2.5 Backend-Design (Soll: Ticket-Plugin)

- Captcha nutzt **bewusst Blau `#2563eb`** („Security/Trust") und die **alte**
  Token-Familie `--bbf-primary/--bbf-*`; Card-Radius 8px; **kein** Drawer, **kein**
  Mobile-Kartenlayout für Tabellen.
- Ticket-Standard: Token-Familie `--bbf-ui-*` (Magenta `#db2e87`, Cyan, Grün),
  Manrope, Gradient-Cards (Radius 18px), Buttons min-height 38px, Fokus-Ring,
  Tabellen→Mobile-Karten via `data-label`, Detail-Drawer, Dark-Mode über
  `html.theme-dark .bbf-plugin-page`, AJAX über `_bbfPost` (jtl_token + JSON/Array).
- → Vollständige Übernahme der `--bbf-ui-*`-Architektur und Komponenten;
  **offener Punkt**: Akzentfarbe (Magenta gemäß Ticket-CI vs. bewusstes
  Security-Blau) — siehe Masterplan §Entscheidungen.

### 2.6 Frontend-Default & Erweiterbarkeit

- Frontend-CSS minimal, **ohne Custom-Properties/Dark-Mode** (feste Farben).
- JS exponiert `window.BBFCaptcha`, aber **keine Events/Hooks/Slots** — heute
  kein Weg, über den AI Concierge, Produktberater oder eine Suche Inhalte
  (z. B. Produktvorschläge) einspeisen könnten.
- Keinerlei Such-/Layer-Infrastruktur vorhanden.

## 3. Risikoeinstufung

| Schwere | Befund |
|---|---|
| 🔴 | Externe Provider ohne Fallback-Kaskade (Kunden-Aussperrung) |
| 🔴 | LLM synchron im Hotpath, kann blockieren |
| 🔴 | Custom-CSS roh injiziert (Injection) |
| 🟠 | Volle IP-Logs ohne Anonymisierung (DSGVO) |
| 🟠 | Auto-Platzierung des Widgets theme-abhängig/unvollständig |
| 🟠 | Keine getestete Versions-/Template-Matrix |
| 🟡 | Backend-Design weicht vom Ticket-Standard ab |
| 🟡 | Kein Dark-Mode/Custom-Properties im Frontend |
| 🟡 | Keine Erweiterungs-API für Fremdplugin-Inhalte |
| 🟢 | Altcha-HMAC-Rotation nur bei Install |
